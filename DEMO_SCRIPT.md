# 演示脚本与实验设计

## 🎬 Demo演示流程

### 演示时长：10-15分钟

---

## 第一部分：背景介绍（2分钟）

### 开场白

"大家好，今天我要展示的是一个**面向跨部门协作的需求转译AI Agent**。

在真实工作中，我们经常会遇到这样的问题：产品经理写完一个需求后，需要分别给研发、设计、算法、测试、运营等不同部门编写不同版本的需求文档。

每个部门关注点完全不同：
- 研发关心功能逻辑、接口、异常
- 设计关心用户流程、页面状态、交互
- 算法关心输入输出、模型边界、评测
- 测试关心测试用例、边界情况
- 运营关心用户价值、指标变化

手动编写这些不同版本既耗时又容易遗漏。这个demo就是用一个AI系统自动完成这个'翻译'工作。"

## 第二部分：Demo演示（8分钟）

### Step 1: 启动应用（30秒）

```bash
./run.sh
```

展示：
- Streamlit界面加载
- 左侧配置栏
- 主输入区域

### Step 2: 选择示例案例（1分钟）

**操作**:
1. 在侧边栏选择"会议纪要AI功能"
2. 展示示例需求内容

**解说**:
"我们先使用一个预设的示例案例：会议纪要AI功能。这是一个典型的产品需求，包含了功能描述、目标用户、使用场景等信息。"

**展示需求内容**:
```
我们希望做一个会议纪要 AI 功能，用户上传录音后，系统可以自动转写、
提炼重点、输出待办事项，还可以生成适合发给老板的简版总结。

核心需求：
1. 支持上传音频文件（mp3, wav, m4a等格式）
2. 自动将语音转为文字
3. 智能识别会议中的关键信息和决策点
4. 自动提取待办事项（谁、做什么、何时完成）
5. 生成不同版本的总结：详细版、简版、行动项版

```

## 第四部分：批判性思考（2分钟）

### 局限性分析

**解说**:
"当然，这个系统也有一些局限性：

1. **LLM可能'编造'技术细节** - 生成的某些技术约束可能不准确，需要人工审核
2. **语气模板化** - 不同部门的语气可能不够贴近真实团队风格
3. **隐性依赖可能遗漏** - 有些隐性依赖LLM可能识别不出来
4. **依赖总控质量** - 如果需求解析不准确，后续所有Agent都会受影响

这些都是未来需要改进的方向。"

### 改进方向

- 引入领域知识库验证技术可行性
- 收集真实团队反馈优化Prompt
- 增加人工审核环节
- 建立历史需求数据库

## 第五部分：Q&A（预留时间）

### 常见问题准备

**Q1: 这个系统和直接让LLM写需求文档有什么区别？**

A: 关键在于Multi-Agent架构。不是让一个LLM生成所有内容，而是：
- 先由总控Agent解析需求
- 再由不同领域的专属Agent从各自视角生成
- 最后由风险Agent综合分析
这样能保证每个版本都符合该部门的专业视角和关注点。

**Q2: 生成的内容准确吗？能直接用吗？**

A: 生成的内容可以作为初稿和参考，但建议：
- 由各部门负责人审核
- 补充团队特定的规范和约定
- 结合实际情况调整
它能节省80%的编写时间，但最后20%的人工审核很重要。

**Q3: 成本怎么样？**

A: 以gpt-3.5-turbo为例，单次转译约$0.03-0.05；gpt-4约$0.50-0.80。相比人工编写多个版本文档的时间成本，这个投入是值得的。

**Q4: 可以集成到现有工作流吗？**

A: 可以的。可以：
- 导出为Markdown/Word集成到Confluence
- 通过API接入Jira创建任务
- 对接企业内部IM发送通知
这些都可以通过扩展实现。

1. **讲故事**: 从一个真实的痛点场景开始

3. **对比展示**: 有vs无，优化前vs优化后
